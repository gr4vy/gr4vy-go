# Testing

This SDK ships an end-to-end test suite that runs against the **E2E sandbox**
(`api.sandbox.e2e.gr4vy.app`). It covers the full merchant transaction lifecycle
and the back-office configuration surface, and reaches every API operation.

> **Note for contributors:** this repository is generated by
> [Speakeasy](https://www.speakeasy.com/). Everything under `test/`, `scripts/`,
> `TESTING.md`, and the CI workflow is **not** regenerated and is safe to edit.
> The shared harness lives in its own `test/harness` package, imported by each
> shard, so an SDK regeneration cannot clobber it.

## Prerequisites

- Go 1.22+ (the CI matrix tests 1.22–1.24).
- The signing key for the E2E sandbox. Provide it either way:
  - Export it: `export PRIVATE_KEY="$(cat /path/to/private_key.pem)"`, **or**
  - Save it as `private_key.pem` in the repo root (git-ignored).

`test/harness` reads `PRIVATE_KEY` first, then falls back to `private_key.pem`.
Without a key the live suites **skip** (so `go test ./...` stays green on a
laptop or a fork), while CI fails loudly if the key is missing on a trusted ref.

## Running the tests

```sh
# Everything (offline unit tests + live E2E)
go test ./...

# A single shard (just point go test at the package)
go test ./test/flows/
go test ./test/processing/
go test ./test/backoffice/

# The offline unit tests (no sandbox needed)
go test . ./optionalnullable/... ./internal/...

# A single test
go test ./test/flows/ -run TestAuthorizeCaptureRefund -v
```

## How it is organised

```
test/
  harness/                 # shared, regen-proof test infrastructure (one package)
    env.go                 # merchant-isolation harness + client builder (PRIVATE_KEY / private_key.pem)
    merchant.go            # TestMerchant: merchant-scoped client + identifiers
    interceptor.go         # forward-compat response injector + HTTP-call recorder
    reach.go               # "endpoint reached + cleanly rejected (4xx)" assertion; fails on 5xx
    fixtures.go            # mock-card inputs, addresses, cart items, unique ids
    checkout.go            # raw PUT /checkout/sessions/{id}/fields + Authorize() helper
    poll.go                # bounded polling for eventually-consistent reads
    gen.go                 # seeded generators for the property-based tests
  flows/                   # headline narratives (transaction + buyer lifecycles)
  processing/              # per-resource CRUD + actions (transactions, refunds, payment methods,
                           #   buyers, gift cards, digital wallets, payment links, checkout sessions)
                           #   + property-based ("uncertainty") tests
  backoffice/              # merchant accounts, payment services, payouts, audit logs,
                           #   account updater, 3DS scenarios, reports
scripts/
  endpoint_coverage/       # builds the operation catalogue from SDK source + reports reach
auth_test.go               # offline JWT tests (package gr4vygo)
webhooks_test.go           # offline signature tests (package gr4vygo)
```

### Isolation & parallelism

Each shard is its own Go package, so `go test` runs it in a separate process.
The harness provisions **one** merchant account (random id) with a `mock-card`
payment service per process — i.e. one merchant per shard — via a package-level
`sync.Once`. Shards therefore share no state and can run fully in parallel. The
shard map lives in `.github/workflows/ci.yaml`, which Speakeasy does **not**
regenerate, so regeneration can't break it.

### The mock-card connector ("test mode")

The deterministic test connector approves card `4111 1111 1111 1111` with a
future expiry and CVV `123` (see `harness/fixtures.go`). Operations that need
real external state (live wallet provisioning, real network tokens, a
payout-capable PSP, a configured 3DS or gift-card service) can't reach a 2xx in
the mock env; those are exercised with `harness.Reaches()`, which asserts the
endpoint was **reached and cleanly rejected with a 4xx**. A **5xx** fails the
test — that means we sent something the API couldn't handle, which is a real
defect to surface.

### Forward-compatibility

The test HTTP client injects a random unknown field into every JSON object
response, proving the SDK tolerates forward-compatible payloads. Disable it for
debugging with `GR4VY_NO_INJECT=1`.

### Property-based ("uncertainty") testing

`test/processing/property_test.go` explores generated inputs (amounts,
currencies, mixed-case metadata) from a fixed seed. Override the seed with
`FC_SEED`; the run count per property is bounded (`harness.Runs`) to keep
live-sandbox cost low. We use a small self-contained generator rather than an
external dependency so the suite stays hermetic.

## Endpoint-reach coverage

When `GR4VY_TRACK_HTTP=1` is set, the test HTTP client logs the method + path of
every request to `coverage/http/*.jsonl`. `scripts/endpoint_coverage` builds the
operation catalogue from the generated resource files (each
`url.JoinPath`/`utils.GenerateURL` path paired with its
`http.NewRequestWithContext` method) and reports how many operations were
reached by a **real HTTP request** — newly generated (but untested) endpoints
show up as "not reached". CI runs this once, un-sharded, and posts the result as
a sticky PR comment. It is a **report, not a merge gate**.

Because each `go test` package runs in its own working directory, the recorder
honours `GR4VY_COVERAGE_DIR` (an absolute path) so every shard writes to one
place; CI sets it. Locally it defaults to `./coverage/http`, so clear that
directory first — otherwise stale logs inflate the reported reach:

```sh
rm -rf coverage/http
GR4VY_TRACK_HTTP=1 GR4VY_COVERAGE_DIR="$PWD/coverage/http" go test ./test/...
go run ./scripts/endpoint_coverage .
```

## CI

- `lint` checks `gofmt`, builds the module, and vets the test suite + tooling.
  (It does not `go vet ./...`: the generated SDK trips vet's unreachable-code
  check, which is upstream's concern.)
- `e2e` runs the three E2E shards on Go 1.24; `offline` runs the offline tests
  on Go 1.22–1.24.
- `ci-complete` is the single stable required check (it `needs` lint + every
  shard) and performs the one-time auto-approve of SDK-regen PRs. Point branch
  rulesets at **`ci-complete`**, not the per-shard matrix jobs.
- `coverage` posts the endpoint-reach comment and is **not** a merge dependency.
- `merge` (auto-merge of bot regen PRs) depends only on `ci-complete`.
